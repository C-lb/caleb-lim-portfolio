# Pitfalls Research

**Domain:** Designer-grade personal portfolio (cross-functional: brand / design / analyst / marketing)
**Researched:** 2026-05-09
**Confidence:** HIGH for AI-tells, motion, PDF embedding, platform lock-in (multiple verified sources). MEDIUM for cross-functional positioning (industry-pattern reasoning, not a single canonical source).

Reference frame: Caleb is non-developer, content is 5-15 static pieces, audience is recruiters skimming for ~30s, design must look hand-crafted (Readymag/Framer references), and he's pitching across four disciplines that recruiters segment by. Every pitfall below is graded against those constraints — not generic web-dev wisdom.

---

## Critical Pitfalls

### Pitfall 1: The "AI-template look" — site silently undermines the brand pitch

**What goes wrong:**
Site ships looking like every Vercel-template / shadcn / lovable / v0 output on the internet. Recruiter for a brand-management role lands on it, pattern-matches it to "generic SaaS landing page generated by an LLM," and updates their prior on Caleb's design judgment downward before reading a word. The pitch ("I'm the brand-creative-with-finance-chops") is contradicted by the medium it's delivered in.

**The visual tells, concretely** (this is what "looks AI-generated" means in 2026 — these patterns are so saturated they read as default-mode):

- **Centered hero**: H1 + subhead + two buttons (primary filled, secondary outline), all stacked center-aligned in the viewport, name spelled out in 60-80px sans-serif.
- **Purple/pink/blue radial or linear gradient** behind the hero, often with a subtle grain/noise overlay. The "AI hero gradient" is its own genre at this point.
- **Inter (or Geist) at every size**, ~16-18px body, tight letter-spacing on headlines. No display face, no serif contrast, no actual typographic personality.
- **shadcn-card grid sections**: three or four equally-sized rounded-corner cards with an icon (lucide-react), a 2-word title, a 1-sentence description. Always three columns on desktop, stacked on mobile. Always with `border border-border bg-card`.
- **Pastel rounded buttons**: `rounded-full` or `rounded-lg`, soft drop shadow, hover gives a 1.02 scale and a brightness bump.
- **Bento grid**: asymmetric-but-still-symmetric tile layout, became the universal "I made an effort" tell after 2024.
- **Tabler/Lucide icons** for everything, monoline 1.5px stroke, 24px.
- **"Built with Next.js" or framework badge** in the footer, or worse, a default Vercel deployment URL.
- **Single accent color** used identically across all CTAs, headings, hover states. No pairing, no system, just one tinted purple/teal/orange.
- **Fade-in-on-scroll** at exactly the same speed and easing on every section (a Framer Motion `whileInView` default that everyone uses unmodified).
- **No real grid or rhythm** — sections are top-padded by the same arbitrary value with no vertical-rhythm logic.

**Why it happens:**
Default-mode taste. The shadcn/Tailwind/v0 stack is genuinely good and everywhere — which is the problem. When the framework's defaults are this strong, "no decision" produces the template look. Owner is non-developer, so any developer building this without explicit anti-template direction will reach for what they ship most often, which is exactly the above.

**How to avoid:**
- Treat reference sites (Readymag editorial pages, Framer expressive portfolios) as a *constraint set*, not inspiration to be loosely riffed on. Pick 3-5 specific reference shots and hold the build accountable to them.
- Pick a **display typeface with personality** for headlines (e.g. a grotesque like GT America/Söhne, an editorial serif like Tiempos/PP Editorial New, or a quirky display like PP Neue Machina) paired with a workhorse text face. Inter-only is the giveaway. Adobe Fonts and Google Fonts both have free options that aren't Inter.
- **No centered hero.** Asymmetric composition: name pushed to a corner, oversized type that bleeds the viewport, type-as-image treatment.
- **No card grids.** Galleries should be irregular — varied tile sizes, intentional whitespace, occasionally a piece that breaks the grid.
- **Real color system**: a primary, a secondary, a neutral palette, an accent — used with hierarchy. Not one purple everywhere.
- **No gradient backgrounds** unless they're deliberate (e.g. an editorial color field, a halftone pattern, a duotone image treatment) — never the default soft-radial-purple.
- **No lucide-react icons.** If icons are needed, draw them, use a less-common set (Phosphor Bold, custom), or skip them and use type.
- **Custom motion**: hand-tune easings, vary timing per section, use motion as a layout primitive (text reveals letter-by-letter, asset rolls in from offscreen) — not as an undifferentiated "fade up" wrapper.
- **Footer must not advertise the stack.** No "Built with Next.js," no "Powered by Vercel," no platform watermark.

**Warning signs:**
- During build, the dev shows a screenshot and you can imagine an identical screenshot on five other sites you've seen this month.
- The first prototype uses Inter for everything.
- The hero section is centered.
- "shadcn" appears in the dependency list. (Not automatically disqualifying — but it means the team has to actively fight defaults.)
- The dev says "I'll just use the default Tailwind config."

**Phase to address:**
**Phase 1 (tech stack & visual direction)** — establish anti-template constraints before any code is written. Write a "design brief / tells to avoid" doc the dev signs onto. **Phase 2 (splash + design system)** — verify: typography is not Inter-only, hero is not centered, no card-grid sections, color system has more than one accent. Re-verify at every milestone.

### Pitfall 2: Cross-functional pitch dilutes into "jack of all trades, master of none"

**What goes wrong:**
Caleb has genuine range — analyst, brand, design, marketing — but a recruiter for a brand-management role wants to know specifically: *can this person do brand management at a competent level?* If the splash and the brand-management gallery don't answer that question with a clear yes, the recruiter assumes "dabbler" and moves on. The other three disciplines, instead of reinforcing range, become evidence of unfocus.

The mis-read happens like this: recruiter sees four equally-weighted disciplines on the splash, infers the candidate spent 25% of their time on each, and discounts each pile to the depth of a hobbyist's. Generalists pay a "specificity tax" on first impression that specialists don't.

**Why it happens:**
- "Show everything" instinct overrides "show enough to convince *this* recruiter."
- Each discipline page is treated as a flat dump of every artifact rather than a curated case for competence.
- Bio reads as a list of skills ("design, finance, marketing, brand") rather than a coherent stance.
- Cross-functional context is told (in copy) but not shown (in artifact quality).

**How to avoke:**
- **Splash framing matters more than copy.** "What do you wish to see?" is the right move because it lets the recruiter declare their context. The four cards have to feel like four legitimate practices, not four hobbies — equal visual weight, professional naming, no cute self-deprecation.
- **Each discipline gallery must lead with the strongest 2-3 pieces.** Not chronological, not random. The first piece a brand recruiter sees on the Design page determines whether they keep scrolling. Lead with the one piece that, if it were the only piece they saw, would still convince them.
- **The 5-15 pieces total split is risky** if it's distributed evenly (≈3-4 per discipline). 3-4 pieces in any one discipline is *thin* by recruiter standards. Better: weight toward the disciplines Caleb is most actively pitching for, accept that the weaker piles have fewer pieces but every piece is strong.
- **Bio takes a stance, not a list.** "I'm a brand-management candidate who does the analyst work too" reads completely differently from "I do brand, finance, marketing, and design." The first claims an identity; the second claims a buffet.
- **Per-piece Context/Role/Outcome blurbs are the specificity proof.** "I designed this poster" is dabbler-coded. "Owned the campaign concept end-to-end, partnered with [team] on production, drove [outcome]" is practitioner-coded. Same artifact, different read.
- **Cross-functional framing is implicit, not explicit.** The site doesn't need a "look at how cross-functional I am!" section — that reads as defensive. The proof is that each individual discipline page holds up on its own merits.

**Warning signs:**
- Discipline pages have wildly different piece counts and the thin ones still aren't curated (e.g. Marketing page has 2 pieces and one is a college project from 2019).
- Bio uses the word "passionate" or "multidisciplinary" or "intersection of."
- Splash copy tries to explain the cross-functional thing instead of letting the user pick.
- A piece appears in multiple discipline galleries (suggests insufficient confidence in which bucket it belongs to).
- The Outcome line in the per-piece blurb is missing or vague ("learned a lot," "great experience").

**Phase to address:**
**Phase 0 / planning (content audit)** — before any build, list every candidate piece, score it within its discipline, ruthlessly cut. Better 3 great pieces in Marketing than 3 great + 4 weak. **Phase 3 (content load)** — writing the Context/Role/Outcome blurbs is content work, not dev work, and is the lever that converts artifacts from dabbler-coded to practitioner-coded. Allocate real time for it. **Bio copy itself is its own milestone** — don't treat it as a 30-minute write.

### Pitfall 3: PDFs and slide decks render badly on mobile (especially iOS Safari)

**What goes wrong:**
Half the recruiter audience is on mobile. Caleb's content is mostly PDF deliverables (slide decks for finance, posters for design). Common implementations break:

- **`<iframe src="file.pdf">`** on iOS Safari renders the PDF as a single static image of page 1, not a scrollable document. This is a long-standing iOS bug going back to iOS 8 — Safari doesn't embed PDFs in iframes the way desktop browsers do; it rasterizes. Recruiter sees what looks like a blurry preview, can't scroll to page 2, leaves.
- **`<embed>` and `<object>` tags** have similar mobile-Safari behavior; not a fix.
- **Google Drive / Dropbox shared-link embeds** require the recruiter to be signed into a Google or Dropbox account that has access — many fail open-link-in-incognito tests.
- **Issuu / SlideShare / Speaker Deck embeds** add their branding, ads, and a clunky viewer chrome around your work.
- **Direct download** works but breaks the gallery flow — recruiter taps a piece, gets a 30MB download dialog, doesn't proceed.
- **PDF.js (Mozilla)** works but adds JS payload and can be slow to first paint on a 4G mobile.
- **Rasterized PDF → image set** (each page exported as PNG/WebP and shown as an image gallery) renders perfectly everywhere but loses text selection and is heavier in bytes if not optimized.

**Why it happens:**
- "Just embed the PDF" is the default mental model and works fine in dev (Chrome on a Mac).
- Mobile Safari testing gets skipped or done late.
- iOS Safari's PDF iframe behavior is quietly broken and not loud about it — the PDF appears to "work" (you see the cover page) but doesn't.

**How to avoid:**
- **Default approach for slide decks: render each page to image at build time.** Astro / Next.js can do this with `pdf-poppler`, `pdf2pic`, or a build-time script using `pdfjs-dist`. Display as a horizontally-paged image gallery (or a vertical scroll-stack for short decks). Provide a "Download PDF" button alongside for recruiters who want the original.
- **Image format**: WebP or AVIF, served at 2-3 sizes via responsive `<img srcset>`. Cap longest edge at ~2000px — slide decks don't need 4K.
- **Fallback for direct PDF**: if a piece must be the PDF itself (resume), link to it directly rather than embedding. iOS will open the system PDF viewer, which works fine.
- **Test on a real iPhone**, not just Chrome devtools' iOS simulation. The simulator doesn't reproduce the iframe-PDF bug.
- **Never use Google Drive embed** as the canonical viewer — link to the PDF as an asset hosted on the site itself.
- **Resume PDF specifically**: keep it under 1MB. ATS systems and email systems sometimes choke on larger files, and recruiters' kneejerk is to email it to themselves.

**Warning signs:**
- Dev demos the site on desktop only and PDFs "work."
- PDF assets in the repo are 10MB+.
- Code uses `<iframe src="*.pdf">` or `<embed type="application/pdf">`.
- A piece's preview looks fine on the gallery page but the detail page just shows a static cover image with no way to see further pages.
- A Drive or Dropbox link is in the JSX as the source of truth.

**Phase to address:**
**Phase 1 (stack pick)** — confirm chosen stack supports build-time PDF-to-image conversion (Astro and Next.js both can; Framer/Readymag have their own asset handling — verify). **Phase 3 (content load)** — every PDF goes through the rasterization pipeline, not as a raw embed. **Pre-launch QA** — test every piece detail page on a real iPhone in Safari, not desktop.

### Pitfall 4: Motion overuse — Framer/Readymag-grade ambition becomes scroll-jacking, content delays, and accessibility fails

**What goes wrong:**
Reference sites (Readymag, expressive Framer portfolios) use motion as a first-class element. The dev tries to match that ambition and overshoots. Symptoms:

- **Scroll-jacking**: scroll wheel input is intercepted; one scroll-tick = one full-section transition (often a CSS-transformed slide). Trackpad users hate it, mouse users hate it, keyboard users (Page Down) often can't navigate it at all. The Webflow accessibility checklist explicitly flags scroll-jacking. WebAIM survey of users with vestibular disorders found ~40% report scrolling problems and >50% specifically named parallax as a trigger.
- **Intro animation gates the actual content**: 2-3 second loading sequence, name-reveal, "enter" button. Recruiter has 30 seconds total — burning 3 of them on animation is 10% of their attention budget gone before anything useful is shown.
- **Cursor-trapping interactions**: oversized custom cursor, magnetic hover effects, cursor that "eats" links and only triggers them on a 200ms hover delay. Looks cool in a portfolio reel, frustrates an actual recruiter.
- **Animation-on-reveal for every element**: text fades up, images fade up, cards fade up — all with 400-800ms delays. The page never feels *there*; it always feels like it's still arriving. On slow connections this stacks badly.
- **Horizontal scroll sections** that hijack vertical scroll. Disorienting, breaks back-button mental model, breaks on touch devices in unpredictable ways.
- **No `prefers-reduced-motion` respect**: users with vestibular disorders, motion sensitivity, or just personal preference get the full ride regardless of OS setting.
- **Heavy JS for motion**: GSAP + Lenis + Framer Motion + ScrollTrigger all loaded together — 200KB+ of JS for a portfolio of 10 pieces.

**Why it happens:**
- Readymag/Framer-style references *look* motion-heavy in screen recordings but in reality use motion sparingly and tactically. Dev sees the highlight reel, not the restraint.
- Motion is fun to build and fun to demo to the client — there's a build-side bias toward more.
- Accessibility is treated as a post-launch consideration, not a build constraint.

**How to avoid:**
- **Motion is a layout primitive, not a wrapper.** Use it where it carries meaning: an oversized name reveals as you scroll past it (carries the editorial tone); a piece's detail page transitions in by sliding the asset to position from where it sat in the gallery (carries spatial continuity). Don't wrap every section in fade-up.
- **No scroll-jacking.** Scroll behavior stays native. Scroll-driven *animation* (using IntersectionObserver or CSS scroll-linked animations to drive a transform) is fine — *intercepting* scroll input is not.
- **No intro animation gates.** First paint shows the splash directly. If there's a brand-stamping moment, it's a 200-400ms flourish that doesn't block interaction.
- **`prefers-reduced-motion: reduce` is honored everywhere.** Replace transforms with instant state changes; replace fade-ups with instant opacity. Test by setting the system preference and verifying no parallax, no scroll-driven movement, no autoplay anywhere.
- **JS budget for motion: ≤50KB gzipped on the splash.** GSAP+ScrollTrigger is ~80KB; Framer Motion is ~50KB; Motion One is ~12KB. Pick one library, not three. For a 10-piece portfolio, Motion One or hand-written CSS animations + IntersectionObserver is plenty.
- **No custom cursor** unless it's genuinely contributing (a paint-stroke trail on the design page that ties to the brand identity, maybe). Default cursor is fine 95% of the time.
- **Touch-first**: every motion should be conceived for touch (no hover dependency, no horizontal-scroll-from-vertical-input).
- **Mobile gets a stripped motion budget** — what works on a 16-inch laptop with 120hz refresh frustrates on a 3-year-old Android.

**Warning signs:**
- "It's like that Readymag site" said in defense of an interaction without specificity about which interaction.
- Lighthouse accessibility score drops below 95.
- The site has a "Loading..." or "Enter" splash before the actual splash.
- Scroll feels heavy or "smoothed" on a regular trackpad.
- Scrolling on iPhone produces unexpected jumps or sticky sections.
- DevTools shows >100KB of motion-library JS on the splash route.
- The dev hasn't tested with `prefers-reduced-motion: reduce` set.

**Phase to address:**
**Phase 1 (visual direction)** — write a one-page "motion brief" that names the 3-5 places motion will live and explicitly excludes scroll-jacking, intro gates, and custom cursors. **Phase 4 (interaction polish)** — verify `prefers-reduced-motion` respected, JS budget hit, real-device tested. **Pre-launch** — Lighthouse accessibility ≥95 and a manual run-through with reduced motion enabled.

### Pitfall 5: Recruiter attention is squandered by friction in the first 30 seconds

**What goes wrong:**
Recruiter has ~30 seconds to decide. Friction patterns that burn that budget:

- **Splash that doesn't communicate the four-discipline concept fast enough.** "What do you wish to see?" is good copy, but if the four cards aren't visually equivalent, instantly readable, and obviously clickable (with discipline names large and clear), the recruiter has to *parse* the splash, which costs seconds.
- **Discipline gallery buries the work below dense copy.** Top of the page is a wall of "About this discipline" text; the grid of pieces is below the fold.
- **Per-piece detail page leads with context, asset is below.** The recruiter taps a piece because they want to *see* the piece. Putting a Context/Role/Outcome wall above the asset makes them scroll.
- **Header navigation is overdesigned** — links so styled they don't read as nav, hamburger that doesn't open instantly, sticky-header that hides on scroll-down (a Medium-era pattern that's universally regretted).
- **No way back to splash from a deep page** other than the browser back button. If a recruiter wants to switch disciplines, they shouldn't have to mentally restart.
- **External links open in same tab** and lose the recruiter; or every link opens in a new tab and they pile up.
- **Slow first paint** — splash takes >2s to render the four cards because of font-loading, motion-library, image lazy-load chain. 2 seconds = 6.6% of attention budget.

**Why it happens:**
- Build is desktop-tested; mobile experience has 1-2s extra latency that gets discounted.
- "Context first, asset second" reads as more thoughtful in design reviews — it isn't, for this audience.
- The splash gets all the design love; gallery and detail pages get treated as "just a list."

**How to avoid:**
- **Splash hits LCP <1.5s on 4G mobile.** Test on throttled connection. Inline critical CSS, preload the display font, don't depend on JS for the four cards to render.
- **Four discipline cards must be readable without parsing.** Discipline name is the dominant element on each card. Hover/tap is obvious.
- **Persistent way back to splash.** Logo or name in header always returns to splash. "Other disciplines" affordance somewhere on every discipline page.
- **Gallery page leads with the grid.** Discipline name and 1-line orientation max above the fold; pieces start above the fold.
- **Detail page leads with the asset.** Large preview first; Context/Role/Outcome below. The blurb is for the recruiter who's already interested, not the gate to becoming interested.
- **External links (LinkedIn, etc.) open in new tab; internal navigation stays in same tab.** Standard but worth verifying.
- **Resume download is one tap from the bio page and from the header.** Don't make the recruiter hunt.

**Warning signs:**
- Splash has a pre-roll, intro text, or anything that delays the four-card choice.
- A recruiter test (have someone unfamiliar look at the site for 30 seconds) results in them not being able to name the four disciplines after.
- Detail page asset is below the fold on mobile.
- Lighthouse mobile performance score below 85.

**Phase to address:**
**Phase 2 (splash + nav design)** — splash is the highest-stakes screen; spend disproportionate design budget there. **Phase 4 (detail page implementation)** — verify asset-first ordering. **Pre-launch UX test** — a 30-second timed run-through with someone outside the project.

### Pitfall 6: Maintenance becomes the developer's job forever

**What goes wrong:**
Caleb is non-developer. Site ships, looks great, works great. Six months later he wants to add a new piece. Options:

- If site is **custom-coded (Astro/Next/etc.) with content-as-files**: he opens GitHub, tries to figure out where pieces are defined, edits a Markdown/JSON/MDX file, commits, hopes the build doesn't fail. In practice this is a high enough friction wall that he doesn't update — site goes stale.
- If site is **on Framer/Readymag**: he opens the visual editor, drags in a piece. Easy. But — if he stops paying the subscription, or the platform shuts down a feature, or pricing changes, the site breaks or becomes hostage. Framer in particular has no HTML export — what's built in Framer stays in Framer (Framer's own docs confirm: published sites depend on backend services like pre-rendering, image resizing, font subsetting; "we can always export later" is not a valid assumption). Plugins exist for CSV/JSON export of CMS data but you don't get the site itself out.
- If site is **on Webflow**: better export story (HTML/CSS/JS export available on paid plans) but still platform-locked for the CMS, and the visual editor learning curve is non-trivial for a non-developer's occasional use.
- If site is **on a CMS-less custom build with a CMS-like admin UI bolted on later**: rarely actually delivered; quoted as "we can add a CMS later" and never gets built.

**Why it happens:**
- Decisions about maintainability get deferred to "later" during the build.
- Developers default to the stack they know, even when that stack is wrong for a non-developer owner.
- Platform lock-in is invisible until you try to leave.

**How to avoid:**
- **Decide explicitly: code path or platform path.** Don't compromise.
  - **Code path** (Astro/Next/Eleventy with content as MDX or YAML/JSON): owner needs a workflow that doesn't require a terminal. Concretely: GitHub.dev (the web editor at `github.dev`) lets you edit content files in-browser; Netlify/Vercel auto-deploy on commit. Document the "add a piece" flow as a 5-step screenshot walkthrough. Test by having Caleb add a piece end-to-end before declaring the site shipped.
  - **Platform path** (Framer recommended over Readymag for this use case based on motion + free tier; Webflow as an alternative): accept the lock-in tradeoff. Pay the subscription, expect to. Keep all source assets (PDFs, images, copy) in a separate cloud folder so you can rebuild on another platform if needed.
- **If code path: content schema must be flat and human-readable.** A piece is one folder with one MDX/YAML file and the asset files alongside it. Not a database, not a CMS schema, not a normalized JSON blob.
- **Whatever path: build the bus-factor-1 doc.** A README in the repo (or a Notion doc) that says exactly: "to add a piece, do X. To update bio, do Y. To swap resume, do Z." Tested by Caleb successfully doing each task.
- **Don't pick a platform for cost reasons alone.** Framer free tier has CMS limits and a `*.framer.app` subdomain — for a real portfolio you're paying $5-15/month minimum. Webflow free tier same story. Cheap host for a custom build (Netlify/Vercel/Cloudflare Pages free tiers) is genuinely free, which makes the code path more attractive *if* the maintenance flow is actually solved.
- **Consider domain registration separately from hosting.** Domain at Cloudflare or Namecheap (~$10-15/yr); hosting separate. If you have to migrate hosts, the domain follows.

**Warning signs:**
- "We'll add a CMS later" in the plan.
- The dev describes the content-update flow using the words "git" or "commit" or "deploy" without a screenshot-level walkthrough that doesn't.
- Free tier of a platform is being used and the dev hasn't checked subdomain/branding/limits implications.
- No bus-factor doc at handoff.
- Caleb hasn't personally added a test piece before launch.

**Phase to address:**
**Phase 1 (stack pick)** — code-vs-platform decision is *the* phase 1 decision and it's downstream of "can Caleb actually update this?" not "what's the most expressive framework?" **Phase 5 (handoff)** — bus-factor doc + Caleb-adds-a-piece dry run before declaring done.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Inline content as JSX/HTML instead of a content file | Fastest to ship pieces during build | Caleb can't update without a dev; every change is a code change | Never — even at 5 pieces, separate content from layout |
| Use a pre-built portfolio template "to save time on layout" | Skips the design hard part | Site looks like the template; AI-tell pitfall guaranteed | Never — undermines the whole pitch |
| Skip mobile testing until the end | Faster desktop iteration | iOS PDF bug, scroll issues, font-size issues compound | Never — test on real iPhone weekly during build |
| Use Google Drive/Dropbox embeds for PDFs | No build-time pipeline needed | Auth required for some viewers; iOS Safari issues; external dependency | Never — host assets on the site |
| Hard-code the four discipline names everywhere | Fastest implementation | Renaming a discipline = touching 5+ files | Acceptable for v1 if Caleb confirms names won't change pre-launch |
| Skip `prefers-reduced-motion` handling | Saves a few hours of motion work | Accessibility lawsuit-bait + actively excludes a chunk of users | Never |
| One JS bundle for everything (no code-splitting) | Simpler build config | LCP suffers on splash; recruiter sees a slow first paint | Acceptable if total bundle <100KB gzipped |
| Skip OG/Twitter card metadata | Faster ship | Site shared on LinkedIn looks like a broken link card | Never — recruiter context, LinkedIn matters |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| PDF embedding | `<iframe src="*.pdf">` or Drive embed | Build-time rasterize to image gallery + separate "Download PDF" link |
| Resume PDF | Embed in-page on About | Direct download link; opens in browser PDF viewer (works on iOS) |
| LinkedIn outbound | `target="_blank"` without `rel="noopener noreferrer"` | Always include both `noopener` and `noreferrer` |
| Email contact | Plain `mailto:` only | `mailto:` + visible email + optional simple form (Formspree/Netlify Forms) for recruiters who don't have a default mail client |
| Analytics | Google Analytics with default config | Plausible or simple-analytics (privacy-respecting, lightweight, no cookie banner needed) |
| Custom domain | DNS configured at domain registrar instead of provider | Use the host's DNS where possible; lower latency, easier to manage |
| Font hosting | `<link>` to Google Fonts (extra DNS, FOUT) | Self-host the font files with `font-display: swap`; preload the display face |
| Image optimization | Ship 4MB JPGs | Astro/Next image components or build-time `sharp`; serve WebP/AVIF with fallback |
| Form submissions | DIY backend | Use Formspree / Netlify Forms / Cloudflare Forms — no server to maintain |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Loading all gallery images at once | Slow first paint on gallery pages, mobile data spike | Lazy-load below-the-fold images with native `loading="lazy"`; preload only the first row | Breaks at gallery >6 pieces on mobile |
| Unoptimized image originals | Pages 5MB+, jank on scroll | Build-time pipeline: WebP/AVIF, multiple sizes via `srcset`, max longest edge ~2000px | Breaks immediately on 4G |
| Motion library bloat (GSAP + Framer Motion + Lenis together) | LCP >2.5s, slow scroll on mobile | Pick one library; for a 10-piece portfolio, Motion One (~12KB) or hand-rolled CSS + IntersectionObserver | Breaks immediately on mid-tier Android |
| Font loading (FOUT/FOIT) | Text invisible for 200-500ms then jumps | `font-display: swap`, preload critical weights, system-font fallback that matches metrics | Breaks first impression every time |
| Client-rendered routing on a static site | Splash flashes content before JS hydrates | SSG with proper static export; check that view-source actually contains the splash content | Breaks SEO + first paint |
| Auto-playing video on splash | Battery drain, mobile data spike, motion sickness | Don't. If video is essential, gate behind a click and respect `prefers-reduced-motion` | Breaks on mobile, every time |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Exposing email as plain text in HTML | Spam scraper bait | Render via JS, use a contact form, or use an obfuscated `mailto:` (`name [at] domain`) |
| `target="_blank"` without `rel="noopener noreferrer"` | Tab-nabbing attack vector | Always pair `_blank` with `noopener noreferrer` |
| Form submissions without spam protection | Bot-filled inbox | Honeypot field + Formspree/Netlify Forms built-in spam filtering |
| Resume PDF contains metadata (author name from MS Word, edit history) | Privacy / unintended info leak | `exiftool -all=` the PDF before publishing, or re-export from a clean source |
| Image EXIF data (location, device) | Privacy leak on personal photos | Strip EXIF in build pipeline (`sharp` does this by default with `withMetadata: false`) |
| Public Git repo with private notes | Old TODO comments, throwaway copy, internal client names leak | Audit repo before making public; better, keep deploy repo private |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Splash that requires interaction before showing the four cards | Recruiter bounces | Cards visible and clickable on first paint |
| Splash that randomizes card order on each visit | Cute, but means the recruiter can't say "click the third one from the left" if they're sharing | Stable ordering; if you want variation, vary visual treatment not position |
| No active-state on the current discipline in nav | Recruiter loses orientation across pages | Active state always indicates current discipline |
| Detail-page back button goes to splash instead of gallery | Recruiter lost their place | Detail page back returns to the discipline gallery they came from |
| Identical-looking pieces in gallery (same crop, same proportion) | Gallery feels like a slideshow, not a body of work | Vary tile sizes, intentional contrast |
| Hover-only affordances (info reveals on hover) | Doesn't work on touch | Make critical info always-visible; reserve hover for delight only |
| Auto-advancing carousel | User loses control, piece they wanted is gone | Manual controls only; no auto-advance |
| 404 page that breaks the brand | Recruiter who got a stale link sees a generic "Not Found" | Custom 404 with link back to splash and a piece of personality |

## "Looks Done But Isn't" Checklist

Verify each before declaring shipped.

- [ ] **Splash on real iPhone Safari**: four cards readable, tap targets ≥44px, no horizontal scroll, LCP <1.5s on 4G throttle
- [ ] **Every PDF piece**: scrollable through all pages on iOS Safari (or rendered as image gallery), download link works, file <5MB
- [ ] **Resume PDF**: opens on iOS Safari, opens on Android Chrome, file <1MB, no embedded metadata leak
- [ ] **`prefers-reduced-motion: reduce`**: site fully usable, no parallax, no scroll-driven motion, no autoplay
- [ ] **Lighthouse mobile**: Performance ≥85, Accessibility ≥95, Best Practices ≥95
- [ ] **OG/Twitter cards**: every URL (splash, four discipline pages, every piece) has a custom share image, title, description — verified in LinkedIn Post Inspector and Twitter Card Validator
- [ ] **Custom 404**: returns 404 status code (not 200), links back to splash
- [ ] **Robots/sitemap**: `robots.txt` allows indexing, `sitemap.xml` lists all pages
- [ ] **Favicon set**: 32x32, 192x192 (Android), 180x180 (Apple touch), .ico fallback
- [ ] **Caleb-update test**: Caleb successfully adds a new piece end-to-end without dev help
- [ ] **30-second test**: someone unfamiliar can name the four disciplines and find one piece in 30 seconds
- [ ] **External links**: open in new tab with `noopener noreferrer`, all targets valid
- [ ] **Email link / form**: actually delivers to Caleb's inbox (test from a different account)
- [ ] **Real-device testing**: iPhone Safari, Android Chrome, desktop Safari, desktop Firefox — not just Chrome
- [ ] **Without JS**: splash and gallery still navigable (graceful degradation, even if motion is gone)

## Recovery Strategies

When pitfalls slip through, how to fix.

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| AI-template look already shipped | HIGH | Visual redesign; not a tweak. Identify 3-5 specific elements (typography, hero, color system) and rework. Don't try to fix it via incremental polish. |
| Cross-functional pitch reads as dabbler | MEDIUM | Audit each discipline page for weakest piece; cut it. Rewrite per-piece blurbs with stronger Role/Outcome lines. Rewrite bio to take a stance. |
| PDF embedding broken on iOS | LOW-MEDIUM | Add build-time rasterization step; replace iframe sources with image gallery. ~1-2 days of work for a 10-piece portfolio. |
| Motion overdone, scroll-jacking shipped | MEDIUM | Strip scroll-jacking entirely (revert to native scroll). Keep motion only on splash and detail page transitions. Re-test with `prefers-reduced-motion`. |
| Slow first paint on splash | LOW-MEDIUM | Inline critical CSS, preload display font, defer motion library, lazy-load gallery images. Lighthouse-driven. |
| Platform-locked on Framer and need to leave | HIGH | Manual rebuild. CMS data exportable via plugins; layout/motion not. Plan: rebuild in code stack using exported CMS JSON as content source. Budget ~2-4 weeks. |
| Caleb can't update site | MEDIUM | Build admin layer if code path (Decap CMS, TinaCMS, or just GitHub.dev workflow with screenshot doc). Or migrate to platform path. |
| OG cards missing/broken | LOW | Add dynamic OG image generation (Vercel/Cloudflare have native handlers) or static OG images per page. ~1 day. |
