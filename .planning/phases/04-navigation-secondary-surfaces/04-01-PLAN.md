      # |-------------------------------|----------|
      # | Input                         | Expected |
      # |-------------------------------|----------|
      # | rel="noopener noreferrer"     | MATCH    |

      Rationale (Pitfall P-7 + boundary correctness): the pattern `([^"]*[[:space:]])?` matches an optional "anything-then-whitespace" prefix before the token, treating the opening quote as the natural left boundary. The suffix `([[:space:]][^"]*)?` matches an optional "whitespace-then-anything" tail before the closing quote. This correctly handles ALL three valid forms — single token (`rel="noopener"`), token at start of a list (`rel="noopener noreferrer"`), and token at end of a list (`rel="noreferrer noopener"`) — while REJECTING comma-separated forms like `rel="noopener,noreferrer"` (modern HTML defines `rel` as a space-separated token list; commas are invalid). The `[[:space:]]` character class is portable across BSD grep (macOS) and GNU grep without relying on `\b` (which BSD grep does not honor reliably inside `-E`).

      Historical note (do not reintroduce): an earlier draft used `(^|[[:space:]])` as the left-side alternative. That form is BUGGY because `^` inside a mid-pattern is a line-start anchor, not a string-start anchor, and `grep -oE` on a single-tag extraction has nothing before the tag to anchor to — so the first token in `rel="noopener noreferrer"` (whose left boundary is the opening quote, not whitespace) never matched. Do not "simplify" the locked form back to that shape.

| origin (caleblim.com) → mail client (user's local MUA) | mailto handoff; not a network boundary, but exposes the inbox to address scrapers when HTML is crawled. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | LinkedIn anchor in Base.astro header (`<a href="https://linkedin.com/in/caleblkr" target="_blank">`) | mitigate | Emit `rel="noopener"` on every `target="_blank"` anchor. Blocks `window.opener.location` rebind from the new tab (reverse tabnabbing). Verified by Gate 20's regex assertion that every `target="_blank"` carries both noopener AND noreferrer (Pitfall P-7 — comma-separated rejected). Modern browsers (Chrome 88+/FF 79+/Safari 12.1+) imply noopener on cross-origin blank targets, but explicit attribute is canonical and survives same-origin edge cases. |
| T-04-02 | Information disclosure | LinkedIn anchor — Referer header leaks the current page URL to linkedin.com on click | mitigate | Emit `rel="noreferrer"` on the same anchor. Strips the `Referer` header on the outbound nav. Verified by Gate 20 (same regex pair). Low-stakes for a public portfolio but documented as the canonical pair with noopener. |
| T-04-03 | Information disclosure | mailto link broadcasts caleblimster@gmail.com to anyone who view-sources or crawls the site (spam scrapers) | accept | Obfuscation (JS-encoded mailto, "click to reveal" pattern) breaks no-JS recruiters and screen readers; the UX cost of breaking direct mail-client launch is higher than the marginal spam reduction. Mailto is published openly — accepted risk. Gmail's spam filter is the operative defense. |
| T-04-04 | Tampering | Resume PDF link in header (`<a href="/caleb-lim-resume.pdf" download>`) — same-origin, no rel needed | mitigate | Keep the resume at the canonical `/caleb-lim-resume.pdf` (same-origin under public/). `download` attribute only fires on same-origin URLs (Pitfall P-5). Future-trap flagged: if anyone moves the resume to a CDN, `download` silently no-ops. No active mitigation needed in Phase 4 — Pitfall P-5 documents the trap for Phase 6 deploy review. |
